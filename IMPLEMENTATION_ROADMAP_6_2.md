# Section 6.2 GBM Benchmark Implementation Roadmap

- Goal:
  - Build a reproducible benchmark pipeline for Section 6.2 that supports GBM simulation, BS delta benchmarking, deep hedging training, multi-objective experiments, scenario grids over `sigma_bar` and `lambda_cost`, multi-seed execution, aggregation, plotting, and publication tables.
  - Keep the implementation modular so the benchmark runner orchestrates work rather than re-implementing math inline.

- Build order:
  - `P0` foundation:
    - `1. simulators`
    - `5. evaluation metrics`
    - shared path/logging helpers used by every later step
  - `P1` single-scenario correctness:
    - `2. hedging baselines`
    - `3. neural policy model`
    - `4. training loop`
  - `P2` benchmark orchestration:
    - `6. experiment runner`
    - `7. aggregation`
    - `8. plotting`
  - `P3` publication outputs:
    - `9. final paper tables`

- Cross-cutting utilities to lock early:
  - Target modules:
    - `src/paths.py`
    - `src/logging_utils.py`
    - `src/hedge_core.py`
  - Inputs:
    - benchmark root directory
    - scenario identifiers: `seed`, `training_regime`, `sigma_bar`, `lambda_cost`, `objective`
    - artifact naming rules
  - Outputs:
    - deterministic run-directory helpers
    - helpers for `run_cfg.json`, `train_log.csv`, metrics files, arrays files, and aggregate paths
    - one canonical path schema used by runner, aggregation, and plotting
  - Dependencies:
    - none beyond the benchmark spec
  - Implement first:
    - `Yes`, alongside Steps `1` and `5`, because unstable paths and schemas create avoidable churn everywhere else

- `1. simulators`
  - Target modules:
    - keep core path dynamics in `src/models_gbm.py`
    - keep dataset construction and normalization in `src/world_gbm.py`
  - Scope:
    - simulate discounted GBM paths
    - create train/validation/test splits
    - support oracle training data and robust mixture training data
    - fit and save feature normalization from the training split only
  - Inputs:
    - `S0`, `K`, `T`, `n`
    - `sigma_true`
    - optional training-vol mixture, for example `{0.15, 0.20, 0.25}` with equal weights
    - `N_train`, `N_val`, `N_test`
    - `seed`
  - Outputs:
    - `t_grid`
    - `S_tr`, `S_va`, `S_te`
    - `Z_tr`, `Z_va`, `Z_te`
    - `F_tr`, `F_va`, `F_te`
    - `feature_norm.json`
    - dataset metadata describing how oracle and robust datasets were assembled
  - Dependencies:
    - payoff function
    - logging/path helpers for saved normalization and metadata
  - Implement first:
    - `Yes`
  - Why first:
    - every later module consumes these tensors or arrays
    - the feature schema must be stable before model and training code are finalized

- `2. hedging baselines`
  - Target modules:
    - keep BS formulas in `src/bs.py`
    - keep pathwise delta construction in `src/strategies_delta.py`
    - evaluate baseline strategies via a reusable evaluator instead of a standalone-only script
  - Scope:
    - compute BS deltas on the fixed hedging grid
    - evaluate baseline performance over `sigma_bar in {0.15, 0.20, 0.25}`
    - compute baseline P&L and turnover under each `lambda_cost`
  - Inputs:
    - `t_grid`
    - `S_test`
    - `Z_test`
    - `K`, `T`
    - evaluation premium `p0`
    - `sigma_bar`
    - `lambda_cost`
  - Outputs:
    - `deltas_bs`
    - `pl_bs`
    - `turnover_bs`
    - `metrics_bs`
    - baseline scenario artifacts and plots
  - Dependencies:
    - simulator outputs
    - shared P&L and turnover math from `src/hedge_core.py` or `src/costs_and_pl.py`
    - metrics functions from Step `5`
  - Implement first:
    - `After Step 1 and the shared P&L interface`
  - Why early:
    - BS delta provides the reference benchmark used to validate the rest of the pipeline

- `3. neural policy model`
  - Target modules:
    - keep the default MLP policy in `src/deep_hedging_model.py`
    - keep rollout logic in `src/hedge_core.py`
  - Scope:
    - define the neural hedge policy interface
    - accept normalized features plus previous hedge position
    - return one delta per hedge date
    - keep architecture configurable without changing the runner
  - Inputs:
    - feature dimension
    - hidden width
    - depth
    - optional activation and delta-clipping settings
  - Outputs:
    - initialized policy model
    - forward pass on one hedge step
    - compatible state dict for checkpointing
  - Dependencies:
    - frozen feature schema from Step `1`
    - rollout contract from `src/hedge_core.py`
  - Implement first:
    - `After Step 1`
  - Why here:
    - the training loop can only be finalized once the policy input and output contracts are stable

- `4. training loop`
  - Target modules:
    - keep runner-independent optimization in `src/train_loop.py`
    - keep objectives in `src/objectives.py`
  - Scope:
    - train one neural hedge model for one `(seed, lambda_cost, objective, training_regime)` cell
    - support multiple objectives behind a common callable interface
    - perform early stopping on validation performance
    - return training history and both best/last checkpoints
  - Inputs:
    - model
    - optimizer
    - objective module
    - training and validation tensors
    - `p0`
    - `lambda_cost`
    - epoch cap, batch size, patience, device
  - Outputs:
    - `best_state.pt`
    - `last_state.pt`
    - `train_log.csv`
    - final validation summary
  - Dependencies:
    - simulator outputs from Step `1`
    - neural policy from Step `3`
    - shared P&L computation from `src/hedge_core.py`
    - objective modules in `src/objectives.py`
    - logging helpers for saved train logs
  - Implement first:
    - `After Steps 1, 3, and 5`
  - Why not before metrics:
    - the loop should emit values in the same schema later consumed by evaluation and aggregation
  - Objective roadmap:
    - first implement:
      - `CVaR / ES` objective at `alpha = 0.95`
    - then extend to:
      - `CVaR / ES` at `alpha = 0.99`
      - entropic-risk objective
      - mean-risk trade-off objective if Section 6.2 needs a secondary robustness study

- `5. evaluation metrics`
  - Target modules:
    - keep scalar risk metrics in `src/metrics.py`
    - keep scenario artifact writing in `src/eval.py`
  - Scope:
    - compute pathwise P&L summaries and turnover summaries
    - define the exact metrics schema used everywhere else
    - save metrics and arrays in a benchmark-friendly format
  - Inputs:
    - `pl_paths`
    - `deltas`
    - `S_test`
    - `Z_test`
    - `lambda_cost`
    - `alpha_list = (0.95, 0.99)`
  - Outputs:
    - `metrics.json`
    - `arrays_debug.npz`
    - scalar metrics:
      - `mean_PL`
      - `std_PL`
      - `VaR_loss_0.95`
      - `ES_loss_0.95`
      - `VaR_loss_0.99`
      - `ES_loss_0.99`
      - `mean_turnover`
      - `max_turnover`
      - `total_turnover`
  - Dependencies:
    - shared P&L and turnover definitions
    - path/logging helpers for stable artifact names
  - Implement first:
    - `Yes`, with Step 1
  - Why first:
    - stable metrics and file schemas are prerequisites for runner logic, aggregation, and plotting

- `6. experiment runner`
  - Target modules:
    - keep the thin single-run training entry point in `src/train_deephedge_gbm.py`
    - add a benchmark orchestrator, for example `src/run_benchmark_gbm.py`
    - optionally add a small config-expansion helper, for example `src/benchmark_grid.py`
  - Scope:
    - expand the benchmark scenario grid
    - run baseline evaluation and deep hedge training for every seed and cost level
    - support oracle and robust training regimes
    - keep per-scenario configs and artifacts isolated in deterministic directories
  - Inputs:
    - base benchmark config
    - seed list
    - `sigma_bar` grid
    - `lambda_cost` grid
    - objective choice
    - model and training hyperparameters
  - Outputs:
    - one run directory per `(seed, strategy, sigma_bar, lambda_cost, objective)` cell
    - copied `run_cfg.json`
    - saved checkpoints, metrics, arrays, and plots
    - benchmark manifest listing all completed scenarios
  - Dependencies:
    - Steps `1` through `5`
    - path helpers for scenario directory creation
  - Implement first:
    - `After a single-seed single-cost oracle run and a matching BS baseline run are already correct`
  - Why here:
    - orchestration should be built on top of verified single-scenario components, not used to debug them

- `7. aggregation`
  - Target modules:
    - add a dedicated module, for example `src/aggregate_benchmark.py`
  - Scope:
    - scan scenario outputs
    - combine seed-level metrics across completed runs
    - compute mean, standard error, confidence intervals, and paired method differences
    - write tidy outputs for analysis and paper tables
  - Inputs:
    - benchmark manifest
    - per-scenario `metrics.json`
    - scenario metadata from `run_cfg.json`
  - Outputs:
    - `seed_level_metrics.csv`
    - `seed_level_metrics.parquet`
    - `scenario_summary.csv`
    - `scenario_summary.tex`
    - `pairwise_tests.csv`
    - paired-difference summaries by `lambda_cost` and comparator
  - Dependencies:
    - stable artifact schema from Steps `5` and `6`
  - Implement first:
    - `After Step 6`
  - Why here:
    - aggregation is only reliable once the runner writes a complete and deterministic set of scenario outputs

- `8. plotting`
  - Target modules:
    - extend `src/plots.py` for reusable low-level plots
    - add benchmark-level plot assembly if needed, for example `src/plot_benchmark.py`
  - Scope:
    - per-scenario plots:
      - P&L histograms
      - tail-metric bars
      - turnover histograms
      - training curves
    - aggregate plots:
      - metric vs `lambda_cost`
      - turnover vs `lambda_cost`
      - paired-difference confidence-interval plots
  - Inputs:
    - per-scenario arrays and metrics
    - aggregate tables from Step `7`
  - Outputs:
    - saved PNG or PDF plots under scenario and aggregate directories
  - Dependencies:
    - metrics schema from Step `5`
    - aggregate summaries from Step `7`
  - Implement first:
    - `After Step 5` for per-scenario plots
    - `After Step 7` for cross-seed plots
  - Why split plotting this way:
    - scenario plots help debug correctness early
    - aggregate plots are only meaningful once multi-seed summaries exist

- `9. final paper tables`
  - Target modules:
    - add a dedicated reporting module, for example `src/paper_tables.py`
  - Scope:
    - convert aggregate benchmark outputs into final dissertation-ready tables
    - separate main tables from appendix sensitivity tables
    - keep publication formatting separate from raw numerical aggregation
  - Inputs:
    - `scenario_summary.csv`
    - `pairwise_tests.csv`
    - plotting outputs and chosen figure references
  - Outputs:
    - main-result LaTeX tables
    - appendix tables
    - possibly a small manifest of figure/table file names used in the dissertation
  - Dependencies:
    - aggregation outputs from Step `7`
    - finalized benchmark reporting decisions from the protocol document
  - Implement first:
    - `Last`
  - Why last:
    - paper tables should sit on top of frozen aggregate outputs, not drive the underlying computation pipeline

- Recommended implementation sequence in practice:
  - Phase `1`:
    - freeze scenario schema, path schema, and metrics schema
    - finish Steps `1` and `5`
  - Phase `2`:
    - make one BS baseline run and one oracle deep hedge run work end to end
    - finish Steps `2`, `3`, and `4`
  - Phase `3`:
    - add the scenario-grid runner with one seed first, then all ten seeds
    - finish Step `6`
  - Phase `4`:
    - implement aggregation and benchmark-level plots
    - finish Steps `7` and `8`
  - Phase `5`:
    - generate the final paper tables and lock the publication outputs
    - finish Step `9`

- Minimal acceptance criteria for each phase:
  - After Phase `1`:
    - one reproducible dataset build writes normalization and metadata correctly
  - After Phase `2`:
    - one single-scenario comparison writes checkpoints, metrics, arrays, and plots
  - After Phase `3`:
    - the full `(seed, sigma_bar, lambda_cost)` grid can be launched reproducibly
  - After Phase `4`:
    - aggregate CSV and LaTeX summaries reproduce the same means and confidence intervals on rerun
  - After Phase `5`:
    - dissertation tables and figures can be rebuilt from saved artifacts without rerunning training
